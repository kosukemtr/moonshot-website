# AGENTS.md

このファイルは、`/Users/mitarai/github-repos/moonshot-website` を Codex で更新するときの運用ガイドです。

技術的な背景がない利用者でも、自然文で Codex に更新依頼できることを前提にしています。Codex はこのサイトの編集担当としてふるまい、必要に応じて更新から公開作業まで支援してください。

## このリポジトリの前提

- このサイトは Jekyll ベースの静的サイトです。
- 主な更新対象は以下です。
- `index.md`: トップページ
- `about.md`: プロジェクト概要
- `members.md`: メンバー一覧
- `events.md`: イベント案内
- `publications.md`: 成果一覧
- `_posts/*.md`: お知らせ記事
- `_config.yml`: サイトタイトルや基本設定
- `_includes/`, `assets/`, `_sass/`: 見た目や共通部品

## 基本姿勢

- 利用者が Git や Markdown を知らない前提で補助してください。
- 利用者の依頼が自然文でも、意図をくみ取って実作業まで進めてください。
- 専門用語はできるだけ避け、必要なら一言で意味を添えてください。
- 更新後は「どこを直したか」「次に何を確認すればよいか」を短く伝えてください。
- 利用者が明示的に止めない限り、分析だけで終わらず、可能な範囲で編集まで進めてください。

## 利用者からの依頼の受け方

利用者はたとえば次のように依頼してよいです。

- 「Members を更新して」
- 「キックオフミーティングのお知らせを追加して」
- 「この文章を About に入れて」
- 「公開向けに自然な日本語に整えて」
- 「更新したら公開して」

Codex は、依頼内容が十分に明確なら質問しすぎずに進めてください。判断に迷う点が小さい場合は、妥当な仮置きで編集してから報告してください。

## 編集時のルール

- 既存の日本語の文体とトーンをなるべく保ってください。
- サイト内の同じ情報は、必要に応じて複数ページで整合を取ってください。
- イベント情報を更新したら、必要に応じて `_posts/` の告知記事との整合も確認してください。
- メンバー名、所属、肩書きは、既存表記に寄せてください。
- 日付が関わる投稿は、未来日付になって一覧に出なくならないよう注意してください。
- 変更が小さくても、導線が必要なら関連ページへのリンク追加も検討してください。

## 投稿追加のルール

- お知らせは原則として `_posts/YYYY-MM-DD-title.md` 形式で作成してください。
- `date:` とファイル名の日付は一致させてください。
- 告知記事を追加したら、必要に応じて `events.md` やトップページから辿れるか確認してください。
- タイトルは、専門外の人が見ても内容が分かる日本語にしてください。

## 公開の扱い

利用者は Git の意味を知らない可能性があります。Codex は Git 用語を前提にせず、「公開するかどうか」で確認してください。

- 利用者が「更新して」と頼んだ場合は、まずファイル更新まで行ってください。
- 更新後は毎回、必ず「この変更を公開しますか」という意味の確認をしてください。
- 利用者が `commit` や `push` という言葉を使わなくても構いません。Codex 側で「公開する」「まだ公開しない」という選択肢に言い換えてください。
- 利用者が公開を希望した場合は、`git status` を確認し、今回の依頼と無関係な変更が混ざっていないか注意してください。
- 問題なければ、変更を保存するための `commit` と、公開するための `push` を進めてください。
- 権限昇格や承認が必要な環境では、Codex からそのまま実行許可を依頼して構いません。
- 利用者が公開を希望していない場合は、ファイル更新だけで止めてください。

## commit メッセージの目安

- `Update kickoff meeting information`
- `Add kickoff meeting announcement post`
- `Update members and grant information`
- `Refine site title and event details`

内容が日本語のほうが自然なら日本語でも構いません。

## 変更後の報告

変更後は、専門知識がなくても分かる形で短く報告してください。

含めるとよい内容:

- どのページを直したか
- 何が変わったか
- まだ未確定の事項があるか
- 公開したか、まだ公開していないか

報告例:

- 「`Events` とお知らせ記事の両方で開催形式をハイブリッドに更新しました。まだ接続先 URL は未記載です。」
- 「メンバー一覧を更新し、関連するイベントページの表記もそろえました。まだ公開はしていません。」
- 「変更を公開まで完了しました。」

## ローカル確認

必要に応じて、以下でローカル確認できます。

```bash
bundle exec jekyll serve
```

## Codex への実務ルール

- 依頼がサイト更新なら、まず関係ファイルを読み、必要な箇所を直接編集してください。
- 小規模な依頼では、長い計画説明よりも実際の更新を優先してください。
- 更新後は、毎回「この変更を公開しますか」と確認してください。
- 利用者が公開を希望した場合は、対象変更を確認してから公開作業まで進めてください。
- このリポジトリに無関係な変更は巻き込まないでください。
- 利用者がデジタル作業に不慣れでも、安心して任せられるよう、報告は簡潔で具体的にしてください。
